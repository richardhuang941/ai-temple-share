# WP02 - Content and Timeline Engine

- 所有对外术语需要与 `claws-temple-bounty2.0-skills` 的品牌词保持一致。

## Subtask Guidance

### T005 - Define TypeScript content models and helper types

**Purpose**

Translate the planning artifacts into concrete TS interfaces so every content module and hook shares one source of type truth.

**Implementation Guidance**

1. Create interfaces/types for:
   - `AgentProfileSnapshot`
   - `TaskMilestone`
   - `TaskStage`
   - `ShareSummary`
   - `AgentPromptCard`
   - timeline state
2. Keep names aligned with `data-model.md`.
3. Prefer explicit union types for task stage statuses and external-flow flags.
4. Do not leak view-specific JSX into these model files.

**Validation**

- [ ] All planned content entities have matching TS types
- [ ] `pending | active | done` is modeled as a strict union
- [ ] The types are reusable by both hooks and components

### T006 - Author hero/share/faction/agent-prompt content modules

**Purpose**

Create the static content source for non-journey sections so UI rendering can stay data-driven.

**Implementation Guidance**

1. Build content modules for:
   - Hero summary and CTA
   - Share summary values
   - Agent prompt cards
   - faction display content
2. Pull wording from the approved spec and planning artifacts.
3. Keep the main visible subject as `Agent`.
4. Use branded public wording; avoid visible-layer forbidden terms like `ca_hash` and `钱包`.

**Validation**

- [ ] Hero copy explains the page in one pass
- [ ] Share content includes score, resonance and faction results
- [ ] Agent prompt data references `Claws-Temple/claws-temple-bounty2.0-skills`

### T007 - Author Task 1-5 milestone and stage data

**Purpose**

Encode the five-task journey into a structured dataset that can drive the simulation UI and the share summary.

**Implementation Guidance**

1. Create one milestone per task, in correct order.
2. For each task, define stage rows that reflect the planned journey:
   - Task 1 score and type result
   - Task 2 identity entry, user ID resolution, resonance success, token gain
   - Task 3 faction choice, oath progression, faction join, Telegram follow-up
   - Task 4 external native flow, publish default, comment alternative
   - Task 5 optional social signal
3. Set flags for:
   - optional task
   - external flow
4. Include proof text that the UI can surface as “completion evidence”.

**Validation**

- [ ] All 5 tasks exist and are correctly ordered
- [ ] Task 4 is flagged as external/native flow
- [ ] Task 5 is flagged as optional

### T008 - Implement content mappers for derived views

**Purpose**

Provide deterministic functions that derive what the UI should show from raw content data and current stage state.

**Implementation Guidance**

1. Implement mapping utilities for:
   - journey card state
   - share card summary
   - current/next task hints
2. Keep these utilities pure and easy to unit test.
3. Avoid embedding component markup here.
4. Ensure share derivation preserves `Agent` wording and joins Task 2 + Task 3 outcomes cleanly.

**Validation**

- [ ] A mapper can derive the public share card state from source content
- [ ] Journey cards receive active/completed/pending states deterministically
- [ ] No component concerns leak into the mapping layer

### T009 - Implement reduced-motion and autoplay timeline hooks

**Purpose**

Create the local state machine that drives the `duangduangduang` sequence and safely degrades when motion should be reduced.

**Implementation Guidance**

1. Implement a hook that tracks current task index and current stage index.
2. Support:
   - autoplay progression
   - manual stepping or restart
   - reduced-motion mode
3. Keep transition rules aligned with `data-model.md`.
4. Expose only the state and handlers needed by later UI WPs.

**Validation**

- [ ] Timeline progresses in order
- [ ] Reduced-motion mode avoids relying on auto-animated state changes
- [ ] Hook output is consumable by journey UI without extra business logic

### T010 - Add unit tests for content mapping and timeline transitions

**Purpose**

Lock the data and state behavior so later UI WPs do not silently regress core journey semantics.

**Implementation Guidance**

1. Add tests for:
   - share summary derivation
   - task ordering
   - Task 4 external-flow preservation
   - Task 5 optional status
   - timeline progression and completion behavior
2. Keep tests small and deterministic.
3. Prioritize business invariants over rendering details.

**Validation**

- [ ] Unit tests cover the core mappings and timeline rules
- [ ] Failures would clearly identify broken business semantics

## Definition of Done

- Content is centralized under `src/content/`
- Timeline logic is isolated under `src/hooks/` / `src/lib/`
- All business wording and task ordering are data-driven
- Unit tests cover mapping and state transitions

## Risks

- If content and UI responsibilities blur here, later WPs will duplicate business logic
- If the timeline hook is under-modeled, Journey UI will need ad hoc state patches

## Reviewer Guidance

- Verify Task 4 remains external/native in the data layer
- Verify `Agent` is the public subject in share-related content
- Verify unit tests assert semantics, not incidental implementation details
