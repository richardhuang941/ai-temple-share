## Subtask Guidance

### T011 - Build stage pulse and status primitives

**Purpose**

Create the smallest reusable journey-specific visual elements that communicate active, done and pending state.

**Implementation Guidance**

1. Build primitives for:
   - active pulse or glow state
   - stage status label
   - completion badge
2. Keep them composable so milestone cards can reuse them.
3. Favor CSS-driven animation over JavaScript-heavy motion.
4. Ensure every visual state still exposes readable text.

**Validation**

- [ ] Active state is visually distinct from done and pending
- [ ] Text labels remain visible in all states

### T012 - Build the per-task milestone card

**Purpose**

Render one task’s branded name, purpose, stage list, proof text and badges in a way that scales from Task 1 through Task 5.

**Implementation Guidance**

1. Use the `TaskMilestone` model as the card’s source.
2. Show:
   - branded task name
   - one-line purpose
   - list of stages
   - current or completed proof text
3. Include visual indicators for:
   - current active task
   - fully completed task
   - optional task
   - external flow task
4. Avoid hardcoding task-specific JSX branches unless absolutely required.

**Validation**

- [ ] A single card component can render any of the 5 tasks
- [ ] Optional and external-flow badges are visible when relevant

### T013 - Build the multi-task journey timeline container

**Purpose**

Assemble all milestone cards into a coherent journey strip that binds to the timeline hook output.

**Implementation Guidance**

1. Render all five tasks in order from content data.
2. Pass current task and current stage state down from the timeline hook contract.
3. Show progression across tasks, not just within a single task.
4. Keep layout adaptable for desktop stacking and mobile linear reading.

**Validation**

- [ ] All 5 tasks render in order
- [ ] Current task changes when the active index changes
- [ ] Completed tasks remain visibly completed

### T014 - Compose JourneySection narrative and task sequence

**Purpose**

Wrap the timeline in the right explanatory copy so the section works as a story, not just a component gallery.

**Implementation Guidance**

1. Add section title, summary and helper copy explaining the journey.
2. Explicitly call out:
   - Task 1-3 are in-path
   - Task 4 is SHIT Skills native flow
   - Task 5 is optional reach
3. Make the section readable even if a user lands here directly from a scroll jump.
4. Keep the tone aligned with the Claws Temple brand voice.

**Validation**

- [ ] Section copy explains why the tasks form a journey
- [ ] Task 4 and Task 5 are correctly framed

### T015 - Make the journey UI responsive and reduced-motion friendly

**Purpose**

Ensure the simulation remains understandable on mobile and accessible when motion is reduced.

**Implementation Guidance**

1. Adapt layout so cards or stages do not force horizontal scrolling.
2. Provide a reduced-motion friendly presentation where state changes can still be followed without pulse-heavy animation.
3. Keep stage text and proof text readable at narrow widths.
4. Prefer graceful degradation over hiding content.

**Validation**

- [ ] No horizontal scroll is required at mobile widths
- [ ] Reduced-motion mode still communicates active state
- [ ] Stage text remains readable on small screens

## Definition of Done

- Journey section renders the five-task path using real content data
- Active/done/pending states are visually and textually clear
- Task 4 external-flow and Task 5 optional status are explicit
- Mobile and reduced-motion behavior are acceptable

## Risks

- Overly flashy animation may reduce readability
- Task-specific hacks may make later copy changes expensive

## Reviewer Guidance

- Verify the section remains readable without relying on animation
- Verify the task order and task purposes match the spec
- Verify Task 4 is not visually disguised as a local closed-loop completion step
