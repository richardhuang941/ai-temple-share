## Subtask Guidance

### T001 - Initialize `React + Vite + TypeScript` runtime and project files

**Purpose**

Create the base project metadata and toolchain files required to run a Vite React TypeScript app in this repository.

**Implementation Guidance**

1. Create or update `package.json` with:
   - Vite dev/build scripts
   - React runtime dependencies
   - TypeScript, Vite and related dev dependencies
2. Ensure TypeScript config files exist and are aligned with a Vite React TS setup.
3. Create `package-lock.json` by installing dependencies with the chosen package manager for this repo.
4. Keep the dependency set minimal; do not add animation or state libraries in this WP.

**Validation**

- [ ] `package.json` contains `dev`, `build`, and `test` scripts
- [ ] TypeScript config files exist and are internally consistent
- [ ] `npm install` succeeds without manual cleanup

### T002 - Configure `Vitest + React Testing Library + jsdom`

**Purpose**

Set up the front-end test harness so later WPs can add unit and integration coverage without reworking the toolchain.

**Implementation Guidance**

1. Add `vitest`, `@testing-library/react`, `@testing-library/jest-dom`, and `jsdom`.
2. Wire a basic `vite.config.ts` test configuration or equivalent test setup.
3. Create `tests/setup.ts` and load `jest-dom` matchers there.
4. Ensure test scripts are ready even if initial tests are added in later WPs.

**Validation**

- [ ] `npm run test` resolves the Vitest runner
- [ ] `tests/setup.ts` is referenced by the test configuration
- [ ] The configuration supports React component testing in jsdom

### T003 - Create global design tokens and base styles

**Purpose**

Lay down the visual foundation for a bold single-page presentation so later UI WPs can inherit a consistent atmosphere.

**Implementation Guidance**

1. Create `src/styles/tokens.css` for color, spacing, type scale, radius, shadows and motion tokens.
2. Create `src/styles/globals.css` for:
   - base reset
   - body background and typography defaults
   - anchor/section spacing
   - utility classes only if clearly reusable
3. Keep the visual direction aligned with the plan: public scorecard + journey poster, not generic SaaS gray panels.
4. Include reduced-motion friendly fallbacks at the global stylesheet level where appropriate.

**Validation**

- [ ] Token file exists and exposes CSS custom properties
- [ ] Global styles include base typography and background treatment
- [ ] No browser-default styling dominates the page shell

### T004 - Prepare the HTML shell and directory skeleton

**Purpose**

Make the repository ready for later section work without forcing later WPs to invent structure ad hoc.

**Implementation Guidance**

1. Update `index.html` with project title and meaningful metadata for the longpage.
2. Create the section/content/hooks/lib component directories needed by the plan if they do not already exist.
3. Do not create business UI files owned by later WPs.
4. Keep the shell focused on bootstrapping and page metadata, not final page composition.

**Validation**

- [ ] `index.html` clearly identifies the project
- [ ] The expected source directories exist for later WPs
- [ ] No files from later WPs are pre-authored here
