# WP03 - Journey Playback and Auto Focus

## Objective

把 Journey 从“自动快进的说明板”升级成“点击才开始、推进更稳、当前任务始终在视野中心附近”的过程模拟区。

## Context

- 用户已经明确要求：不要默认自动执行，必须点击 `观看模拟 Task 1-5 的流程` 后再开始。
- 当前节奏偏快，需要拉长间隔。
- 当前任务推进时，页面必须滚到对应 task 附近，保证用户能直接看到进行到哪一步。
- reduced-motion 环境下不能强制平滑滚动，但仍要保证当前任务聚焦逻辑成立。
- Journey 展示的业务语义要消费 WP01 的真实 task 数据，而不是硬编码新解释。

## Subtask Guidance

### T009 - Slow the playback cadence and keep autoplay opt-in only

**Purpose**

Make the simulation feel deliberate and readable rather than like a rapid-fire status flash.

**Implementation Guidance**

1. Ensure the initial timeline state does not auto-start on first render.
2. Keep the existing CTA-driven entry path intact.
3. Increase the interval between stage advances to a more readable pace.
4. Preserve restart and manual advance semantics where useful.

**Validation**

- [ ] Journey remains idle before the user triggers it
- [ ] Playback cadence is visibly slower than the previous version
- [ ] Restart behavior still works

### T010 - Add reduced-motion-safe auto-focus

**Purpose**

Move the viewport with the story, but only when the story meaningfully advances.

**Implementation Guidance**

1. Add a focus strategy that reacts to task changes and only the most meaningful stage changes.
2. Use native scroll behavior and guard against repeated no-op scrolling.
3. In reduced-motion mode:
   - avoid forced smooth scrolling
   - still bring the active task into view
4. Keep the focus logic isolated enough to test or reason about.

**Validation**

- [ ] Current task is brought into the main viewport when the story advances
- [ ] The page does not visibly jitter on every minor tick
- [ ] Reduced-motion mode still preserves focus correctness

### T011 - Refine journey cards for visibility and semantics

**Purpose**

Make each task card tell the user what just happened, what proof exists, and whether the flow continues externally.

**Implementation Guidance**

1. Emphasize the active task clearly against completed and upcoming tasks.
2. Surface proof lines and external-action hints in a way users can scan quickly.
3. Keep Task 4 visibly marked as a native handoff.
4. Make Task 5 look optional rather than mandatory.
5. Favor a strong card-state hierarchy over decorative noise.

**Validation**

- [ ] Active task state is obvious
- [ ] Proof and external-flow cues are easy to spot
- [ ] Optional vs required tasks are visually distinct

### T012 - Add unit coverage for timeline and focus behavior

**Purpose**

Protect the new flow rules so future UI tweaks do not silently reintroduce autoplay-on-load or overactive scrolling behavior.

**Implementation Guidance**

1. Add tests for:
   - idle initial state
   - start/restart behavior
   - slower progression assumptions where practical
   - task advancement boundaries
2. If focus logic is factored into pure helpers, cover those helpers directly.
3. Prefer behavior assertions over DOM snapshots.

**Validation**

- [ ] Timeline tests fail if autoplay sneaks back into initial render
- [ ] Focus rules are testable without brittle render snapshots

## Definition of Done

- Journey starts only from the dedicated CTA.
- Playback cadence is slower and easier to follow.
- Current task stays visible as the simulation advances.
- Reduced-motion behavior remains accessible and predictable.

## Risks

- If focus logic is mixed directly into render effects without guardrails, the page may jitter or repeatedly steal scroll.
- If the active-state styling is too subtle, the new focus behavior will still feel weak.

## Reviewer Guidance

- Confirm that the simulation does not start itself on page load.
- Confirm that the active task reliably moves into view during playback.
- Confirm that Task 4 still reads as a native handoff and Task 5 still reads as optional.
