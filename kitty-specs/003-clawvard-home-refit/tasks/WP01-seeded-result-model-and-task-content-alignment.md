# WP01 - Seeded Result Model and Task Content Alignment

## Objective

把这轮 refit 的“事实层”钉住：同一轮页面里，Hero、Share、Journey 必须引用同一组 score / grade / type / axes / faction 结果；同时 `Task 1-5` 的叙事必须和本地 bounty skill 完全对齐，而不是沿用旧的占位式解释。

## Context

- `Task 1` 不能只剩一个打分，要体现评分结果、等级、类型、六轴倾向和 faction 映射。
- `Task 2` 需要体现身份入口、登录恢复/首次进入、用户 ID 解析、开放寻配与共振完成。
- `Task 3` 需要体现 faction 选择、正式 oath、`2 AIBOUNTY` 门槛、加入阵营和 Telegram follow-up。
- `Task 4` 不是站内完成态，它是 SHIT Skills 的 native handoff，默认推荐 `publish`，但要保留 `comment` 语义。
- `Task 5` 是 optional 社交信号，不能压过主线资格。
- 这个 WP 只负责数据、类型和 mapper，不负责 Hero/Journey UI 排版。

## Subtask Guidance

### T001 - Build a deterministic result seed

**Purpose**

Create a small seed utility that produces one coherent public result per browser session, so the page can feel “randomized” without becoming contradictory.

**Implementation Guidance**

1. Add a dedicated utility module for the seeded result instead of burying randomness inside components.
2. Generate and persist a session-scoped value that can deterministically produce:
   - score
   - grade
   - agent type label
   - dominant axes
   - faction mapping result
3. Keep the output bounded to realistic ranges for the challenge card.
4. Make the seed logic easy to test without `window` coupling when possible.

**Validation**

- [ ] One seed always produces one stable result set
- [ ] Faction result is derived from the same seed, not a second random source
- [ ] The API is consumable by pure mapping functions

### T002 - Expand content models and mapper outputs

**Purpose**

Upgrade the typed content layer so public surfaces can render the same seeded result with localized copy and no hidden per-component business logic.

**Implementation Guidance**

1. Extend the model layer for seeded result fields the UI now needs:
   - grade badge
   - faction summary
   - challenge blurb fragments
   - task proof values
2. Refactor mapper outputs so downstream sections receive already-derived public data rather than recomputing from raw content.
3. Keep locale-sensitive templates data-driven.
4. Avoid moving JSX or DOM concerns into mapper code.

**Validation**

- [ ] Hero, Share, and Journey can all derive their display data from the same seeded source
- [ ] Locale-aware challenge strings remain template-driven
- [ ] Mappers stay pure and deterministic

### T003 - Rewrite Task 1-5 data against local skill semantics

**Purpose**

Replace the stale simulation copy with a dataset that actually reflects the local skill repository and faction config.

**Implementation Guidance**

1. Update `Task 1` to surface score, grade, type, axes, and faction mapping proof.
2. Update `Task 2` to reflect the identity entry and resonance flow rather than a vague registration success story.
3. Update `Task 3` to include:
   - faction selection
   - approval or prerequisite check
   - formal oath
   - `2 AIBOUNTY` threshold fact
   - joined result
   - Telegram follow-up
4. Keep `Task 4` clearly marked as native and external, with `publish` as the default recommendation and `comment` as fallback.
5. Keep `Task 5` explicitly optional and framed as social amplification.

**Validation**

- [ ] Task ordering stays 1 through 5
- [ ] Task 3 visibly references the token threshold
- [ ] Task 4 remains a native handoff instead of fake completion
- [ ] Task 5 remains optional

### T004 - Add unit coverage for seed consistency and mapping invariants

**Purpose**

Lock the new result model so later UI refits cannot reintroduce inconsistent or incorrect business semantics.

**Implementation Guidance**

1. Cover deterministic seed output and same-session stability.
2. Cover share/hero/journey mapping consistency for score / grade / faction.
3. Cover task semantics invariants:
   - Task 3 threshold copy present
   - Task 4 external/native semantics preserved
   - Task 5 optional flag preserved
4. Keep the tests narrow and business-focused.

**Validation**

- [ ] Failing tests would clearly reveal a broken public result contract
- [ ] The tests do not depend on section markup

## Definition of Done

- Seed generation is deterministic and reusable.
- Content models and mappers express the new public-result contract.
- Task data reflects the local skill semantics.
- Unit tests cover the core seeded-result and task-semantics invariants.

## Risks

- If the seed contract is under-modeled here, later sections will quietly fork their own result logic.
- If the task dataset still paraphrases from memory instead of the local skill repo, the public demo will remain semantically wrong.

## Reviewer Guidance

- Verify the same result fields power Hero, Share, and Journey.
- Verify Task 3 includes the token threshold language.
- Verify Task 4 still points outward and never claims local completion.
